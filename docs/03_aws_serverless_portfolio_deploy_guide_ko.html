AWS Serverless Portfolio 배포 이해 노트

서버리스 포트폴리오 배포 명령을 “어디서, 왜, 무엇을 만드는지” 중심으로 일반 예시값으로 정리한 HTML 버전.

프론트 React build 파일을 S3에 저장하고 CloudFront로 HTTPS 제공
도메인 도메인 등록기관에서 산 example.com를 Route 53이 관리
백엔드 API Gateway가 Lambda의 /health, /profile 호출

→ https://example.com
→ Route 53
→ CloudFront
→ private S3 bucket
→ 프론트엔드 정적 파일

Frontend app

1. AWS CLI 프로필 설정

실행 위치

어디서 실행해도 된다.

명령

# YOUR_AWS_PROFILE이라는 이름으로 AWS CLI 로그인 정보 저장
aws configure --profile YOUR_AWS_PROFILE
AWS CLI가 어떤 IAM 사용자 권한으로 AWS에 접근할지 저장하기 위해서.
Access Key IDIAM 유저의 access key
Secret Access KeyIAM 유저의 secret key
Default regionYOUR_BACKEND_REGION
Outputjson

확인

# 현재 profile이 어떤 AWS 계정/유저로 연결되는지 확인
aws sts get-caller-identity --profile YOUR_AWS_PROFILE

2. 백엔드 API 배포

실행 위치

cd /path/to/your-serverless-portfolio/backend

명령

# Python 가상환경 켜기
source ../.venv/bin/activate

# Lambda 배포 준비물 생성
sam build

# Lambda + API Gateway를 AWS에 배포
sam deploy --guided --region YOUR_BACKEND_REGION --profile YOUR_AWS_PROFILE
프론트엔드 앱이 호출할 서버리스 API를 만들기 위해서. 이 단계에서 API Gateway와 Lambda가 생성된다.

guided 입력값

Stack Nameyour-portfolio-api-stack
AWS RegionYOUR_BACKEND_REGION
Confirm changesY
Allow IAM role creationY
Disable rollbackN
No authentication okay?y
Deploy changeset?y

결과

PortfolioApiBaseUrl이 생성됐다.

https://YOUR_API_ID.execute-api.YOUR_BACKEND_REGION.amazonaws.com/Prod

확인

curl https://YOUR_API_ID.execute-api.YOUR_BACKEND_REGION.amazonaws.com/Prod/health

정상 응답: {"status":"ok","service":"your-portfolio-api"}

3. Route 53 Hosted Zone 생성

명령

# example.com의 DNS 관리 공간 생성
aws route53 create-hosted-zone \
  --name example.com \
  --caller-reference example-com-UNIQUE_TIMESTAMP \
  --profile YOUR_AWS_PROFILE
도메인 등록기관에서 산 도메인의 DNS 관리를 AWS Route 53에서 하기 위해서.

생성된 값

HostedZoneIdYOUR_HOSTED_ZONE_ID

도메인 등록기관에 등록한 네임서버

ns-XXXX.awsdns-XX.co.uk
ns-XXXX.awsdns-XX.org
ns-XXXX.awsdns-XX.com
ns-XXXX.awsdns-XX.net

의미: 도메인 등록기관에게 “이제 example.com DNS 관리는 Route 53이 한다”고 알려준 것.

4. ACM HTTPS 인증서 생성

명령

# CloudFront에 붙일 HTTPS 인증서 요청
aws acm request-certificate \
  --region us-east-1 \
  --domain-name example.com \
  --validation-method DNS \
  --profile YOUR_AWS_PROFILE
https://example.com로 접속하려면 인증서가 필요하다. CloudFront용 ACM 인증서는 us-east-1에 만들어야 한다.

인증서 ARN

arn:aws:acm:us-east-1:YOUR_ACCOUNT_ID:certificate/YOUR_CERTIFICATE_ID

5. ACM DNS 검증

검증 레코드 확인

aws acm describe-certificate \
  --region us-east-1 \
  --certificate-arn arn:aws:acm:us-east-1:YOUR_ACCOUNT_ID:certificate/YOUR_CERTIFICATE_ID \
  --profile YOUR_AWS_PROFILE \
  --query "Certificate.DomainValidationOptions[0].ResourceRecord"

Route 53에 CNAME 추가

aws route53 change-resource-record-sets \
  --hosted-zone-id YOUR_HOSTED_ZONE_ID \
  --profile YOUR_AWS_PROFILE \
  --change-batch '{
    "Changes": [{
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "_YOUR_ACM_VALIDATION_NAME.example.com.",
        "Type": "CNAME",
        "TTL": 300,
        "ResourceRecords": [{
          "Value": "_YOUR_ACM_VALIDATION_VALUE.acm-validations.aws."
        }]
      }
    }]
  }'
AWS가 DNS를 조회해서 이 CNAME이 있으면 “도메인 DNS를 수정할 권한이 있네”라고 판단하고 인증서를 발급한다.

상태 확인

aws acm describe-certificate \
  --region us-east-1 \
  --certificate-arn arn:aws:acm:us-east-1:YOUR_ACCOUNT_ID:certificate/YOUR_CERTIFICATE_ID \
  --profile YOUR_AWS_PROFILE \
  --query "Certificate.Status"

최종 상태: ISSUED

6. 프론트 호스팅 인프라 생성

실행 위치

cd /path/to/your-serverless-portfolio

명령

# S3 + CloudFront + Route 53 record 생성
aws cloudformation deploy \
  --region us-east-1 \
  --profile YOUR_AWS_PROFILE \
  --stack-name your-frontend-hosting-stack \
  --template-file infra/frontend-hosting.yaml \
  --parameter-overrides \
    DomainName=example.com \
    HostedZoneId=YOUR_HOSTED_ZONE_ID \
    AcmCertificateArn=arn:aws:acm:us-east-1:YOUR_ACCOUNT_ID:certificate/YOUR_CERTIFICATE_ID
프론트엔드 사이트를 담을 “집”을 만드는 단계다. 아직 파일 업로드가 아니라 S3 bucket, CloudFront, Route 53 alias를 만드는 작업이다.

생성된 값

BucketNameYOUR_FRONTEND_BUCKET_NAME
CloudFrontDistributionIdYOUR_CLOUDFRONT_DISTRIBUTION_ID
WebsiteUrlhttps://example.com
CloudFrontDomainNameYOUR_DISTRIBUTION.cloudfront.net

7. 프론트엔드 앱 빌드

실행 위치

cd /path/to/your-serverless-portfolio/frontend

명령

# 실제 API Gateway 주소를 프론트엔드 앱에 넣어서 production build
VITE_API_BASE_URL=https://YOUR_API_ID.execute-api.YOUR_BACKEND_REGION.amazonaws.com/Prod npm run build
Vite는 VITE_API_BASE_URL 값을 빌드 시점에 JS 안에 넣는다. 그래서 실제 API 주소가 생긴 뒤 다시 빌드해야 했다.

결과 폴더: frontend/dist/

8. S3 업로드와 CloudFront 캐시 삭제

S3 업로드

# dist 폴더 내용을 S3 bucket에 동기화
aws s3 sync dist s3://YOUR_FRONTEND_BUCKET_NAME --delete --profile YOUR_AWS_PROFILE
빌드된 프론트엔드 정적 파일을 S3에 올려야 CloudFront가 사용자에게 보여줄 수 있다.

CloudFront 캐시 무효화

# CloudFront가 새 파일을 바로 보도록 전체 캐시 무효화
aws cloudfront create-invalidation \
  --distribution-id YOUR_CLOUDFRONT_DISTRIBUTION_ID \
  --paths "/*" \
  --profile YOUR_AWS_PROFILE
CloudFront는 이전 파일을 캐시할 수 있다. 캐시 무효화를 해야 새로 올린 프론트엔드 파일이 바로 반영된다.
